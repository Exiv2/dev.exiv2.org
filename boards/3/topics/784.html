will 0.22 appear in time for spring releases of major Linux distributions?

Added by Adam Dingle almost 11 years ago

Hello -

I work on the popular photo manager Shotwell, which uses libexiv2 extensively. A number of our users have been affected by an exiv2 bug (http://dev.exiv2.org/issues/752) which Andreas fixed just yesterday. Thanks very much for the fix!

Unfortunately, without this fix Shotwell crashes for some users - see http://trac.yorba.org/ticket/3104 . So it would be great to get this fix out to users. Specifically, it would help us a lot if you could release a new exiv2 0.22 in time for it to be included in the spring releases of the major Linux distributions, namely Ubuntu 11.04 and Fedora 15. The feature freeze for Ubuntu 11.04 is on February 24 (see https://wiki.ubuntu.com/NattyReleaseSchedule), so I think it would be easiest to make it into that Ubuntu if exiv2 0.22 could appear in the next few weeks. Might that be realistic? (If not, maybe we can work with Ubuntu to cherry-pick this particular fix, but it would be better if they could simply take a new exiv2 release.)

By the way, thanks for creating an incredibly useful library!

adam

RE: will 0.22 appear in time for spring releases of major Linux distributions? - Added by Andreas Huggel almost 11 years ago

I've uploaded a patch for bug #752 for version 0.20 and linked to the patch for 0.21 in the issue itself.

It should be possible to release a binary compatible version 0.21.1 with all the fixes applied so far if we don't include the recent Windows specific changes, and an incompatible release 0.22 with these Windows changes shortly after. I'll create a branch in svn for 0.21.1 with the goal to release it by mid February. If for some reason I won't find the time for the release, there will be at least the 0.21.1 branch which the distros can use.

Mike and Adam, can you please contact the concerned maintainers and check if that plan works for them?

Regarding the issue itself, I'm curious what causes the makernote corruption that triggers the Exiv2 bug. Clearly the sample image attached to the issue has been modified by some other software. The Exiv2 bug has been present for a long time, but apparently was never triggered until recently. So has a new release of a popular software recently started to corrupt Canon makernotes like this? Can you check with your users what programs they used to process their images?

Andreas
