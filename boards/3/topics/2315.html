RE: How to get Pentax ISO - Added by Tobias E. about 6 years ago

After trying all possible ways to get a value out of an Exifdatum I was reminded that sometimes write() behaves differently than toString() or any of the other methods (you should think about that API for a while), so I used the following code in the end:

// ...

std::ostringstream os;
pos->write(os, &exifData);
std::string os_str = os.str();
const char *exifstr = os_str.c_str();
img->exif_iso = (float)std::atof(exifstr);

Maybe it's helpful for others in the future.

I've looked a little more at this and now understand what you're describing. '39' is the value stored in Exif.PentaxDng.ISO. 39 is an index into the table at extern const TagDetails pentaxISO[] = { .. } in src/pentaxmn.cpp. 

661 rmills@rmillsmbp:~/gnu/exiv2/trunk $ exiv2 -pv --grep ISO YIMGP2303.DNG 
0x8827 Photo        ISOSpeedRatings             Short       1  65535
0x0014 PentaxDng    ISO                         Short       1  39
0x0071 PentaxDng    HighISONoiseReduction       Byte        2  255 4
662 rmills@rmillsmbp:~/gnu/exiv2/trunk $
So toLong() and toFloat() return 39. The method print() returns the 'translated' value: 
662 rmills@rmillsmbp:~/gnu/exiv2/trunk $ exiv2 -pt --grep ISO YIMGP2303.DNG 
Exif.Photo.ISOSpeedRatings                   Short       1  65535
Exif.PentaxDng.ISO                           Short       1  204800
Exif.PentaxDng.HighISONoiseReduction         Byte        2  (255 4)
663 rmills@rmillsmbp:~/gnu/exiv2/trunk $
It looks like std::atof(exifdata->print()) is the simple solution.

As pentaxISO is an extern, you might be able to link and reference that table from your code.

Yes, it's not elegant to reference something in an Internal namespace and if the library's internals are changed, your code will break. Better to copy the table into your code.

Yes, these are the correct APIs to get the "interpreted" or "pretty-printed" value. The documentation explains a bit more. Note that the Metadatum output operator can also be used to achieve the same thing.
And no, the internal lookup tables are not meant to be accessible, I hope you won't succeed with that. (IIRC they are marked external for some template magic to work)
