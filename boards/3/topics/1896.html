RE: Using multiple commands. So how? - Added by Robin Mills over 6 years ago

I'm not sure you can do this, Chris. The syntax is:

exiv2 [options] [action] file ...

The ellipsis ... refers to multiple files. So you can perform the same action(s) on multiple files. For example:

663 rmills@rmillsmbp:~/temp $ exiv2 -M"set Exif.Image.Model this-is-b-you-know" A.jpg B.jpg
$ exiv2 -pa -g Model A.jpg B.jpg
A.jpg                 Exif.Image.Model                             Ascii      19  this-is-b-you-know
B.jpg                 Exif.Image.Model                             Ascii      19  this-is-b-you-know
$
You may perform more than one modification on one or more files.
$ exiv2 -M"set Exif.Image.Model RobinModel" -M"set Exif.Image.Make RobinMake" A.jpg
$ exiv2 -pa -g Exif.Image.M A.jpg
Exif.Image.Make                              Ascii      10  RobinMake
Exif.Image.Model                             Ascii      11  RobinModel
$
However, I don't see any way to perform different actions on different files in a single invocation of exiv2.exe. If I attempt this:
$  exiv2 -M"set Exif.Image.Model RobinModel" A.jpg -M"set Exif.Image.Make RobinMake" A.jpg
Exiv2 exception in modify action for file -Mset Exif.Image.Make RobinMake:

We get an exception because the argument -M"set..." is being parsed by the syntax file... and of course the file: -Mset Exif.Image.Make RobinMake: does not exist.

The library can perform any number of modifications on any number of open files, however I don't believe the syntax of the exiv2 command provides different commands for different files. To achieve that, you'll have the use the dentist's method (you pull them out one at a time). The DOS syntax for this would be:

exiv2 -M"...." C:\path-to-file1 && exiv2 -M"different" C:\path-to-file2 && exiv2 -M"another change" C:\path-to-file3
There is obviously a performance hit to invoke exiv2.exe. However the library requires no initialization, we parse the command-line then process the files one at a time. The library uses memory mapping when available, (which it normally is), and processes the meta-data with efficiency. It does not read the image into a frame buffer.
