RE: Write exif metadata without temporary file? - Added by Andreas Huggel over 12 years ago

You have a good point here. For TIFF images, the temporary is really not needed (*): TiffImage::writeMetadata() reads the whole image into memory first, then encodes the metadata and finally writes it back to a file. But for the large images that you have, that's also not what you want I guess. Do you see a better approach? ()

(*) For other image formats, the algorithm is different and the temporary is really needed, JPEG for example.

Writing is a 2-pass process: 1st, Exiv2 attempts to update the existing structure in-place ("non-intrusive" writing). That works if no data value is added or deleted or grows in size. E.g., if a timestamp was modified. It is fast and has the advantage that it preserves the complete TIFF structure even if Exiv2 doesn't know it in detail. 2nd, only if non-intrusive writing is not possible, it will re-write the TIFF structure.

Maybe I can do the 1st step directly in the file, and don't have to read it... That would potentially limit the required resources dramatically.

Andreas

RE: Write exif metadata without temporary file? - Added by Andreas Huggel over 12 years ago

I've opened bug #617 and experimented with updating the memory mapped image directly, which has a considerable effect as expected. The next step will be to replace the Blob used in the TiffParser with a BasicIo. With this, it will be possible to write the image directly to a file instead of to another copy in memory for intrusive writing.

The backdraw of the change in r1755 is that in case Exiv2 crashes somewhere in the non-intrusive writing logic, this is now likely to result in a corrupted image.

Profiling will be needed to find out what other measures to take, I suspect there is more potential (for 0.18 I was happy that it could write TIFF images in the first place :)

However, all of this may not address your use-case within a reasonable time. Writing 4GB images is not exactly a common scenario and chances are that Exiv2 still copies the image data somewhere. A more practical approach might be to hide the image data from Exiv2: With Exiv2, you could compose all the required TIFF tags - including the placeholder image data tags but without the actual image data - and write them to a (still small) file. With the size of the file, the offsets to the image data is then known, so you can update the offsets in the image tags (non-intrusively this time). Then you append the actual image data at the end. Possibly this approach might require a slight customization of Exiv2, to treat the TIFF image tags as ordinary tags (just remove them from the TIFF structure table).

Andreas
