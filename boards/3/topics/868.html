JPEG Comments null termination

Added by Ian Binnie over 10 years ago

I used eviv2 to set some JPEG Comments and a trailing null was added to the comment.
This is probably not an error, as the content of these comments is undefined, but it is different from every other application I have used, and it is inconvenient, because it causes display errors in some programs.

I wrote a simple program to copy JPEG Comment to the IPTC Caption, which worked, but the library re-wrote the original JPEG Comment with a trailing null, which seems unacceptable.


Replies (3)

RE: JPEG Comments null termination - Added by Andreas Huggel over 10 years ago

Exiv2 actually strips off all trailing nulls when it reads the comment and appends one when it writes it back. That seems to have always been like this.
On the other hand, the wrjpgcom sample program that ships with libjpeg doesn't append a trailing null (checked versions 6b and 8c).

So the exiv2 behaviour is indeed questionable, maybe it should simply leave the comment as it is.

Andreas

RE: JPEG Comments null termination - Added by Ian Binnie over 10 years ago

I changed jpgimage.cpp line 769 to
us2Data(tmpBuf + 2, static_cast<uint16_t>(comment_.length() + 2), bigEndian);

This seemed to fix the issue as it reduced the size of the Comment field.

The code at line 772 which actually writes the comment does not seem to explicitly write the null, only comment_.length(), so the null must just come from the buffer initialistion.
