Problems with unicode metadata

Added by Anonymous almost 13 years ago

Hi,

I’m having slight problems reading Unicode metadata from some files.
We have customers using a variety of languages (Hebrew and Russian amongst them).
We’re using the Exiv2 library to parse metadata from images as they are imported into our database.
For the most part this works fine, the images with Hebrew metadata work fine but I’ve been given an image from a Russian client which is failing to import correctly.

I’ve run the images through the command line version of exiv2.

I’m running on Windows XP; there’s a setting in the “Regional and Language Options” control panel to set the default language for non-unicode languages.

The output for the image with Hebrew metadata looks okay if set the default language to Hebrew.
For the image with Russian text, I can do the same but only some of the fields look correct. When I read them in through the API, I’m getting the same problem. Some fields are in Cyrillic character set, some are not.

If I open either document in Adobe Bridge, which is the software being used by the Russian client, then the fields all look like regular Russian character set (as far as I can tell).

My guess is that some fields are in UTF8 and some are not, but how the heck can I tell which is which when I’m reading them in through the API. Somehow, Adobe are managing to get it right so the information must be there somewhere.

I enclose a sample image with Hebrew and Russian metadata, as well as my output text file from the command line.

Any clues or help greatly appreciated.

RE: Problems with unicode metadata - Added by Andreas Huggel almost 13 years ago

Hi Caerwyn,

Some general comments, I haven't looked at your images yet.

As a rule, for now at least, the character set in which metadata is
encoded is transparent to Exiv2.

Different types of metadata (Exif, XMP, IPTC) may be encoded
differently in an image, thus you may observe inconsistent encodings.
For Exif and IPTC values, Exiv2 doesn't do any character set
conversions, except for a few specific Windows XP Exif tags (#460,
supported if compiled with iconv enabled - which isn't the case on
Windows by default). Strings from metadata are read from an image
and output without any conversion.
For XMP, Exiv2 also doesn't do any conversions, but the XMP-SDK (XMP
Toolkit from Adobe) used to parse and serialize XMP data may. The
XMP-SDK API always returns and expects strings in UTF8, thus XMP
strings that you get through the Exiv2 API are in UTF8. However,
according to the XMP specification, XMP data in JPEG and TIFF images
(and several other image formats) must be encoded in UTF8, so no
conversion is required. (I'm not sure what XMP-SDK does if this is not
the case.)

The Metadata Working Group has published a technical specification
with useful information on the topic. Exiv2 will eventually have to
become more aware of character sets, along the lines of the these
guidelines.
There are two open bugs for Exiv2 saying that at least in certain
cases Exiv2 should care about character sets: #562 and #571.

Andreas
