RE: Using exiv dll in MFC (VC++ 6.0) application - Added by Robin Mills over 10 years ago

Shilpa

We don't support MSCV v6.0 for several reasons including:
1) Nobody's ever requested this.
2) C++ template support in MSVC6.0 had poor standards compliance. So it probably won't compile, even if I/you constructed a good build environment.
3) MSVC is rather old (about 1998, I think).
4) We support many MSVC versions (Express/Standard/Pro) 2003 (32 bit only), 2005, 2008 and 2010 (32 and 64 bit).

We have CMake support and I've used it with some success on MSVC 2003 and up. I plan to support CMake/MSVC later this year (for the 0.23 release). CMake does have support for MSVC6.0, however simply fixing the build environment won't fix the template/compilation issues with MSVC6.0

The Express Edition of MSVC 2010 is available free from microsoft.com. I believe can successfully create a 32 bit of the libraries. I recommend using the msvc64 build environment with MSVC 2010 (even for 32 bit builds).

You've mentioned that you have built the libraries with MSVC 2008. Is there a reason why you are unable to build your application with 2008 ?

Robin

RE: Using exiv dll in MFC (VC++ 6.0) application - Added by Robin Mills over 10 years ago

Shilpa

I think it's unlikely that you'll be able to link the MSVC 2008 lib/dll to an app built with MSVC6.0. The reason is that the compiler generates code that refers to the run-time support libraries, and so you'll probably find that the library either doesn't link or won't run. However it's quite easy to try an experiment using the exiv2/sample/exifprint.cpp source code. It is possible that it works!

The alternative is (as you have suggested) to build/download exiv2.exe and invoke it from your MSVC6.0 application (using CreateProcess or something). You might find that exiv2.exe displays a DOS box - in which case you can probably use SrvAny from the Windows resource kit to run exiv2 as a service (which runs on an invisible desktop). If you're calling exiv2 a lot, you might prefer to build a "proper" service and avoid the overhead of starting/terminating exiv2 every time you want to use it. You can communicate with a service using a named pipe or the message-queue, sockets or some other IAC mechanism. However before you spend a lot of effort on this - try using CreateProcess to invoke exiv2.exe - that might be sufficient for your needs.

I have a copy of MSVC6.0 at home. Please let me know if you want help with any of this.

Robin

RE: Using exiv dll in MFC (VC++ 6.0) application - Added by Robin Mills over 10 years ago

Shilpa

I believe this is possible, however I don't recall ever building exiv2 without expat, xmpsdk and zlib support.

There's a file exiv2\msvc\include\exv_msvc.h which contains directives which you can set. Additionally, you'll have to change the project settings to remove explicit linking of the (sub) libraries and project dependencies. I've just tried this and although I've been able to link without zlib (which provides PNG support), I did get unsatisfied link errors requiring XMP code. So I build and linked expat and xmpsdk (but not zlib).

A static Release build of exiv2.exe (on version 0.21 32 bit build) is only 1.4mb - so he's not fat even if he contain code you don't really want.

Robin
