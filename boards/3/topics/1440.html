GSoC 2013 (Google Summer of Code)

Added by Robin Mills almost 9 years ago

In 2012, the Exiv2 project enjoyed having a GSoC student join us. GSoC is "Google Summer of Code". Students are generously sponsored by Google to undertake open source projects during their University summer vacation/recess.

The project added support for 4 video formats to Exiv2 and was undertaken by Abhinav Badola, supervised/mentored by Andreas and Gilles. The guru of all meta-data, Phil Harvey said in advance: "I feel sorry for the poor soul assigned to this task. Video metadata is a real hairball.". http://dev.exiv2.org/issues/813

Abhinav added the following code:

And I'm sure there were additional code changes elsewhere in the code base to support this effort.

Shawn and I became involved late in the project and submitted small changes (about 20 lines, I think) to persuade everything to build, run, pass the test suite using MSVC (32 and 64 bit), Cygwin and Clang.

From my point of view, I observed:

  1. The code was neat, tidy, documented and readable
  2. He's a pleasant and agreeable person with whom to talk and work
  3. He totally finished the job
    • added code to the test harness
    • took advantage of the forum to collect 50mb of test files from users
    • supported the porting to MSVC, Cygwin and Clang.

Thank You, Abhinav. Great Work.

_______________________________________________________________________________

And now it's 2013 and Gilles has asked "What about GSoc 2013?"

I'm willing to run the treadmill of KDE/Google (essential) bureaucracy. However we have to nominate and define a project. The purpose of this topic is to discuss possible projects. In the spirit of open-source, the team has agreed to discuss this openly on the forum. Anyone may contribute ideas, suggestions, and criticism if they wish. Possible projects are:

  1. Support for more video formats
  2. Support for WebP (and/or WebM) format
  3. Better CMake support (especially for MSVC)
  4. I/O support for web protocols (http in particular)
  5. Investigate strategies to harness more exiftool Perl-based technology into exiv2 C++ codebase
  6. Remote build server
We shouldn't discuss students in this thread for a couple of reasons:
  • The application process is Google's business
  • We shouldn't discuss individuals in public without their prior knowledge and consent

And finally, I'd like to suggest that we "fork" additional threads from here to discuss individual projects in detail.

Project Proposal - JSON and Scripting Support

5 years ago, I became interested in exiv2 to implement a GeoTagging application. I decided to use Python as an excuse to learn the language. I used the pyexiv2 wrapper, written by Olivier, and the project was a success. Building exiv2 and pyexiv2 on Windows and MacOSX was a challenge (to say the least).

Since then, I've worked steadily on the exiv2 msvc and msvc64 built environments and I believe both are working very well.

Sadly, building pyexiv2 is remains a challenge because it requires boost and the scons build utility. (scons is/was another GSoC project.) The consequence is that my python script seldom uses the latest exiv2 and is not available on all my machines (Windows/Cygwin/Mac/Kubuntu). The script is stable (hardly been changed in 5 years), however building the pyexiv2 wrapper is a maintenance challenge. The pyexiv2 has to be built for specific versions of python (2.6, 2.7 etc), architecture (32/64 bit), platform (windows/cygwin/macosx/linux).

This is not a criticism of Olivier's pyexiv2 wrapper. Olivier has done a very good job. Python wrappers which link C++ are a severe maintenance challenge. I haven't worked for years with Perl's C++ support (XS and/or SWIG), however I anticipate similar pain and trouble.

JSON to the rescue. My proposal is to provide a JSON interface to read and write meta-data in the exiv2 command-line utility.

As an sample application to prove our JSON support, provide wrappers for Perl and Python. The wrappers can be written entirely in the scripting language and use the language's JSON support. There is no need to get involved with C++ integration challenges such as boost/scons/pyexiv2, xs and swig. When reading from files, the wrapper will call exiv2.exe ONCE to capture all JSON to file. When writing to files, the wrapper will call exiv2.exe ONCE. This strategy will enable the wrappers to and run on all platforms on which exiv2.exe is available.

Proposal: I/O support for web protocols (http in particular)

Today we provide support files available on the file system. These files can be memory mapped if this feature is supported by the host OK.

With the increasing interest in "cloud" computing, it's become ever more common for files to reside in remote locations which are not mapped to the file system. Very common cases today are ftp and http. For example: http://bla/bla/bla/file.jpg. Today there are a myriad of "Cloud" storage products, such as AWS, DropBox, Google Drive, Sky Drive, Box,iCloud, Just Cloud and more.

The proposal is to support http, ftp and ssh. This can be done by creating new instance of the BasicIO abstract class. The exiv2 command would accept filenames with a URL (in addition) to the local file system. For example:

exiv2 -pt http://clanmills.com/files/Robin.jpg
exiv2 -pt ftp://username@password:/clanmills.com/Robin.jpg
exiv2 -pt ssh://username@password:/clanmills.com/Robin.jpg

In most image files, the meta-data is defined in the first 100k of the file, so the implementation should only read blocks on demand from the server and avoid copying the complete file. It's easy to envisage mobile applications to display an image preview and metadata, such as image size, data and GPS location. Efficient protocol support can deliver performance and data at modest band-width consumption. Another use case for this technology is a "metadata-server" web-service. By supplying the URL of the image, the service can return metadata without copying the whole image.

The simplest possible implementation of this proposal for exiv2 to detect the protocol and use a helper application such as curl or ssh. This implementation probably requires copying the complete file from the remote storage to a temporary file in the local file system. While such an implementation can be constructed quickly, this does not satisfy the project aim to make efficient use of band-width. It is very desirable to use a robust implementation of the web protocols and a library such as libcurl should be considered. The selection of the protocol support library must respect build implications. We should be careful to avoid adding a large library (such as boost) to the build dependancies. Additionally, the implementation is required to be written in C++ and run on Mac/Windows/Linux without dependency on platform frameworks such as .Net, Java, or Cocoa. It may be that build switches can be provided to enable Exiv2 to use platform frameworks. This could be especially useful on mobile platforms such as Android and iOS.

The implementation should provide bi-directional support (both read and write) with read-access being the first priority.

Finally, an engineering assessment of the effort and technology to support "Cloud" storage (such as AWS and DropBox) should be part of this effort. This assessment would provide a starting point for a future GSoC project.

RE: GSoC 2013 (Google Summer of Code) - Added by Robin Mills almost 9 years ago

Indeed the protocols nfs: and smb: could be supported. However because these protocols enable the user to map remote objects into the filesystem, support in exiv2 for those protocols doesn't provide unique/new functionality.

An interesting extension is to enable exiv2 to run as a service (daemon) on a web socket. I imagine two types of clients:
  1. exiv2 itself of course
  2. JavaScript/WebSocket client

To do this we could do something like this:

Server:      # exiv2 --daemon --port 54321
Client:      $ exiv2 -pt exv://server:54321:/Robin.jpg
Even better: $ exiv2 -pt exv://server:54321:/http://clanmills.com/files/Robin.jpg
I don't want to get into detail concerning the JavaScript API for this. Something like this:
<script src="js/Exiv2.js">
var exiv2     = new Exiv2( { server : 'clanmills.com' , port : 54321 }); 
var metadata  = eval(exiv2.command('--JSON -pt /Robin.jpg '));
// or even better
var metadata  = eval(exiv2.command('--JSON -pt http://clanmills.com/files/Robin.jpg'));
To get the most from this functionality, we should provide JSON (and/or XML) support which I discussed in an different proposal.

However, a project to extend remote I/O support to take advantange of WebSockets seems like a future GSoC project. That's one idea in my head when I proposed an engineering assessment of other "Cloud" technologies.

RE: GSoC 2013 (Google Summer of Code) - Added by Abhinav Badola almost 9 years ago

Proposal to add a new interface to Exiv2, to help add new lens.

Many a times I have seen users and developers reporting missing lens information. They then have to post it on forum, or open up an issue under Exiv2. Then they have to wait for the developers to update the code so that they can proceed with their work and testing.

It would be really great if we could add an interface to Exiv2, where it would allow users to add new lens information to their local Exiv2 installation.

I mean if we could add some interface where user can temporarily add information to a local database and then access the information through Exiv2, as if the Lens information was actually existing in Exiv2.
This would temporarily facilitate the developer/user to continue on with their work, while we can work on the issue and resolve it.

[1] We can create a separate text file, where the user can add Lens information in a predefined structured form.
or
[2] We can add some new command line parameters like --addlens
eg.-
exiv2 --addlens --id 255 --makernote "Canon" --lensname "Tokina New Lens To Be Added"

Most of the times, our users are developers or experienced coders, and they directly edit the code and add the lens information themselves, in their local repository.
But this feature can help them save a lot of editing and compiling time as well.
